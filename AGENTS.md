# AGENTS.md

## Overview

Privacy-first video editor, with a focus on simplicity and ease of use.

## Lib vs Utils

- `lib/` - domain logic (specific to this app)
- `utils/` - small helper utils (generic, could be copy-pasted into any other app)

## Core Editor System

The editor uses a **singleton EditorCore** that manages all editor state through specialized managers.

### Architecture

```
EditorCore (singleton)
├── playback: PlaybackManager
├── timeline: TimelineManager
├── scene: SceneManager
├── project: ProjectManager
├── media: MediaManager
└── renderer: RendererManager
```

### When to Use What

#### In React Components

**Always use the `useEditor()` hook:**

```typescript
import { useEditor } from '@/hooks/use-editor';

function MyComponent() {
  const editor = useEditor();
  const tracks = editor.timeline.getTracks();

  // Call methods
  editor.timeline.addTrack({ type: 'media' });

  // Display data (auto re-renders on changes)
  return <div>{tracks.length} tracks</div>;
}
```

The hook:

- Returns the singleton instance
- Subscribes to all manager changes
- Automatically re-renders when state changes

#### Outside React Components

**Use `EditorCore.getInstance()` directly:**

```typescript
// In utilities, event handlers, or non-React code
import { EditorCore } from "@/core";

const editor = EditorCore.getInstance();
await editor.export({ format: "mp4", quality: "high" });
```

## Actions System

Actions are the trigger layer for user-initiated operations. The single source of truth is `@/lib/actions/definitions.ts`.

**To add a new action:**

1. Add it to `ACTIONS` in `@/lib/actions/definitions.ts`:

```typescript
export const ACTIONS = {
  "my-action": {
    description: "What the action does",
    category: "editing",
    defaultShortcuts: ["ctrl+m"],
  },
  // ...
};
```

2. Add handler in `@/hooks/use-editor-actions.ts`:

```typescript
useActionHandler(
  "my-action",
  () => {
    // implementation
  },
  undefined,
);
```

**In components, use `invokeAction()` for user-triggered operations:**

```typescript
import { invokeAction } from '@/lib/actions';

// Good - uses action system
const handleSplit = () => invokeAction("split-selected");

// Avoid - bypasses UX layer (toasts, validation feedback)
const handleSplit = () => editor.timeline.splitElements({ ... });
```

Direct `editor.xxx()` calls are for internal use (commands, tests, complex multi-step operations).

## Commands System

Commands handle undo/redo. They live in `@/lib/commands/` organized by domain (timeline, media, scene).

Each command extends `Command` from `@/lib/commands/base-command` and implements:

- `execute()` - saves current state, then does the mutation
- `undo()` - restores the saved state

Actions and commands work together: actions are "what triggered this", commands are "how to do it (and undo it)".

## Reproducing the Editor with Real Pipeline Data

To open the editor pre-loaded with a real drama project (multiple scenes, tracks, subtitles) instead of a blank project, use the import flow.

### Source of truth

Pipeline output JSONs (`opencut_{episode_id}.json`, OpenCut v10 `SerializedProject`) are generated by the **Platform** repo (`Auto_Video_Editor`, see its `public/scripts/`). They are NOT committed into OpenCut-AI.

### Manual import (dev)

1. Pick a pipeline JSON, e.g. `Auto_Video_Editor/public/scripts/opencut_<id>.json`.
2. Copy it into OpenCut-AI's static folder so it is served on the same origin (avoids CORS): `apps/web/public/<file>.json`.
3. Start the web app (`bun run dev:web`, port 3001) and open:
   `http://localhost:3001/editor/<any-id>?import=http://localhost:3001/<file>.json`
4. The import flow runs automatically:
   - `app/editor/[project_id]/page.tsx` fetches the JSON, post-processes audio, stores it in `localStorage["opencut:pending-import"]`, then redirects to `/editor/<json.metadata.id>`.
   - `components/providers/editor-provider.tsx` → `importProjectPhase1` writes the project to IndexedDB, then redirects again.
   - On the final load, `importMediaPhase2` creates placeholder media (colored PNGs per shot via `addMediaAsset`) and normalizes each scene: `startTime` (absolute → per-scene 0), `fontSize` (pipeline px → editor units, `px * 90 / canvasHeight`), text `transform.position` (absolute → center-relative), and audio (`media-tts-*` → Platform library URLs).

### Platform dependency (audio)

Audio / TTS tracks reference the Platform: `http://localhost:3000/assets/audio/tts/<id>.mp3`. If the Platform (Auto_Video_Editor, port 3000) is **not** running, those tracks show "Audio unavailable" and WaveSurfer/`mediabunny` emit `Failed to fetch` errors. This is expected when testing OpenCut-AI in isolation — start the Platform for full audio. The errors are environmental, not OpenCut-AI bugs.

## Performance: `useEditor()` is a monolithic subscription

`useEditor()` subscribes to **all** managers (playback, timeline, scenes, project, media, renderer, selection) and re-renders on **any** notify. With many consumers (e.g. 51 `DraggableItem`s in the assets panel) a single `notify()` re-renders the whole React tree including `ResizablePanelGroup`, which is expensive (Issue #235).

Rules established by Issue #235:

- **Continuous playback must NOT call `playback.notify()`** per frame. See `PlaybackManager.updateTime()` — it only dispatches the `playback-update` CustomEvent at 60fps and keeps `notify()` for state transitions (play/pause/seek/boundary).
- Real-time consumers read time **without React**: the preview canvas reads `getCurrentTime()` inside its `useRafLoop`; the timeline playhead and the preview timecode subscribe to the `playback-update` window event and update the DOM / local state directly.
- Do not re-introduce a `notify()` (even throttled) inside the playback time-tick — it re-creates the UI lag. If a component needs live time, use the `playback-update` event, not `useEditor()`.
